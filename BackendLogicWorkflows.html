schema Developer Reference

Backend logic workflows for the provider and member portals.

This page documents the backend orchestration behind the portal experience. It is meant for developers, architects, and delivery teams. The portals are the interaction layer for members and providers to use AI and legacy-friendly flows. Auto-adjudication is already assumed to exist upstream, so the workflows below focus on intake, orchestration, routing, enrichment, state transitions, integrations, notifications, and audit.

Primary Actors

Member, Provider, Nventr, Human Ops

Key Modes

Prompt, Voice, File, Workflow Routing

Runtime Assumptions

1. Auto-adjudication already exists. The portal does not replace adjudication logic. It prepares, enriches, routes, tracks, and explains work around adjudication.

2. Portal is an orchestration surface. Members and providers use the UI, AI assistant, voice prompts, and file uploads to trigger backend workflows.

3. AI is assistive and operational. Nventr can collect missing fields, classify intent, summarize artifacts, package workflows, and escalate to human teams.

4. Legacy flows remain available. Even when AI can perform the action, users can still move through the classic form-by-form experience.

Core Backend Layers

Portal API Gateway

Session bootstrap, role-based routing, request normalization, auth context, throttling.

Workflow Orchestrator

State machines, task routing, retries, compensating actions, event emission.

Domain Services

Enrollment, claims intake, prior auth, appointments, pharmacy, billing, follow-up, notifications.

AI + Human Handoff Layer

Prompt handling, ambient listening, summarization, file extraction, escalation packets.

Workflow Catalog

Backend workflows that sit behind the portals and AI actions.

WF-01

Portal access and role bootstrap

Used when a visitor picks Provider or Member on the landing page and enters sign-in or first-time setup.

unfold_more

Backend flow

  1. Persist selected role context and requested portal entry point.
  2. Create anonymous session or signed-in session bootstrap record.
  3. Resolve feature flags, role policies, tenant, and AI settings.
  4. Route user either to provider verification, member enrollment, or the correct dashboard.

State keys

  • role_context
  • session_id
  • entry_flow
  • dashboard_route

AI role

Suggest next best route, explain first-time access, and prepare the right workflow shell before the user reaches the next screen.

WF-02

Member enrollment and activation

Registration, KYC, plan mapping, eligibility, payment setup, dependent support, and dashboard activation.

unfold_more

Backend flow

  1. Create member_enrollment_draft on registration.
  2. Call KYC service for ID and identity validation.
  3. Fetch plan data using member ID, sponsor, or direct plan selection.
  4. Run eligibility verification through payer connectivity.
  5. If required, tokenize payment method and persist billing setup reference.
  6. Activate member record, generate member ID, and emit activation event.
  7. Provision dashboard, notification preferences, and dependent-management capability.

Primary services

  • Enrollment Service
  • KYC Service
  • Eligibility Service
  • Plan Mapping Service
  • Payment Profile Service

Output events

  • member.enrollment.saved
  • member.eligibility.verified
  • member.activated
  • member.preferences.updated

WF-03

Provider verification and workspace activation

Provider first-time setup before access to claims, authorizations, payments, appointments, pharmacy, and members.

unfold_more

Backend flow

  1. Create provider verification case.
  2. Store demographic and credential metadata.
  3. Validate uploaded license or ID against provider registry or credentialing source.
  4. Activate provider workspace permissions and org affiliation.
  5. Bootstrap queue counts, dashboard metrics, and AI workspace context.

Queues

  • provider_verification_review
  • provider_activation
  • provider_workspace_bootstrap

AI role

Explain required uploads, summarize verification state, and prepare a human review packet if verification confidence is low.

WF-04

Claim submission and pre-adjudication orchestration

Claims are prepared, enriched, routed, and tracked here. Auto-adjudication remains external and already established.

unfold_more

Backend flow

  1. Capture claim draft, selected CPT and ICD codes, supporting files, and submitter context.
  2. Run AI enrichment: code recommendation, missing attachment checks, denial-risk flags, revenue-integrity checks.
  3. Persist normalized claim packet and create claim work item.
  4. Submit packet to clearinghouse or claim intake gateway.
  5. Track intake status, acknowledgements, edits, and downstream auto-adjudication references.
  6. Surface status, appeal eligibility, and follow-up tasks back into the portal.

Services

  • Claims Intake Service
  • Coding Assist Service
  • Revenue Integrity Service
  • Attachment Store

Output events

  • claim.saved
  • claim.submitted
  • claim.status.changed
  • claim.appeal.ready

WF-05

unfold_more

Backend flow

  2. Run AI extraction for diagnosis, procedure, urgency, and missing medical-necessity evidence.
  3. Generate payer packet and e-form payload.
  4. Submit to authorization endpoint or admin queue.

Human-in-the-loop points

Key statuses

  • Validating
  • Submitted

WF-10

AI prompt, voice, file, ambient listening, and human handoff

The cross-cutting workflow that turns Nventr into an execution surface instead of just a chatbot.

unfold_more

Backend flow

  1. Accept prompt source: typed, voice transcript, uploaded files, or ambient listening transcript.
  2. Classify intent: claims, auth, payment, appointment, enrollment, pharmacy, care journey, support.
  3. Extract entities and required evidence.
  4. Either answer directly, launch a workflow, or create a human handoff packet.
  5. Persist conversation metadata, workflow correlation ID, and audit record.

AI artifacts

  • Intent classification
  • Entity extraction
  • Summaries and action plan
  • Workflow launch command

Handoff packet

  • Conversation summary
  • Files and references
  • Suggested next action
  • Confidence and unresolved questions

Detailed Feature Workflows

Feature-by-feature backend logic for implementation planning.

Each workflow below connects a visible prototype feature to the backend trigger, orchestration steps, services, state, events, and escalation paths that engineering will need. These are intentionally implementation-facing so product, architecture, and delivery teams can review scope from one page.

FW-01

Access, sign-in, role routing, and demo reset

Covers landing page persona selection, returning sign-in, first-time setup, role switching, and consolidated demo reset.

unfold_more

Trigger

User selects Provider or Member, signs in, chooses first-time setup, or resets demo state.

Backend steps

  1. Create or resume session with tenant, persona, auth status, and requested entry point.
  2. Evaluate role policy, onboarding state, enrollment state, and provider verification state.
  3. Route provider to verification or provider dashboard; route member to enrollment resume or member dashboard.
  4. Emit route decision and update last-accessed workspace.

Services and state

  • IAM, Role Policy, Session Bootstrap, Demo State Service.
  • session_id, role_context, onboarding_status, dashboard_route.
  • Events: access.role.selected, access.signin.completed, access.route.resolved.
  • Fallback: if state is ambiguous, route to role selector with a safe explanation.

FW-02

Persistent AI shell, prompt routing, and workflow queue

Covers Nventr assistant, suggested prompts, workflow launch, action queue, page-aware context, and legacy fallback.

unfold_more

Backend steps

  1. Receive prompt, page context, role, user state, and any attached workflow references.
  2. Classify intent into claims, prior auth, payments, appointments, enrollment, pharmacy, care journey, or support.
  3. Validate required entities and request missing fields conversationally when needed.
  4. Queue a workflow command or return an explain-only response if mutation is not needed.
  5. Persist audit trail with source page, prompt text, confidence, and correlation ID.

Services and events

  • AI Action Service, Workflow Command Bus, Prompt Policy, Audit Service.
  • Events: ai.prompt.received, ai.intent.classified, workflow.command.queued.
  • State: conversation_id, intent, entity_refs, action_status.
  • Human checkpoint: low confidence, PHI-sensitive mutation, or user-requested escalation.

FW-03

Voice prompts, ambient listening, call-center AI, and live agent

Covers voice capture, ambient intake, transcript summarization, call resolution, and live-agent handoff readiness.

unfold_more

Backend steps

  1. Capture consent, input channel, language, device context, and session correlation ID.
  2. Stream or upload audio to transcription service with PHI-safe retention policy.
  3. Summarize transcript, classify call intent, and extract member, claim, auth, payment, or scheduling references.
  4. Create an intake case and route to AI resolution, callback queue, or live specialist.
  5. Store final transcript summary, unresolved questions, and SLA metadata.

Services and controls

  • Voice Gateway, Speech-to-Text, Call Intent Classifier, Agent Queue, SLA Service.
  • Events: voice.capture.started, ambient.summary.ready, handoff.case.created.
  • Controls: consent required, transcript retention class, escalation threshold, queue ownership.
  • Output: call summary, recommended next action, assigned queue, and user-facing status.

FW-04

File-based AI actions and document lifecycle

Covers claim documents, prior auth packets, appeal evidence, statements, EOBs, referrals, and lab or imaging files.

unfold_more

Backend steps

  1. Upload file to secure object store and create document metadata record.
  2. Run malware scan, file-type validation, OCR, classification, and PHI tagging.
  3. Extract structured fields and attach document references to the active workflow.
  4. Run confidence checks and route low-confidence extraction to human review.
  5. Apply retention policy and maintain immutable access audit.

Services and state

  • Document Store, OCR, Extraction, PHI Tagger, Attachment Policy, Audit.
  • Events: document.uploaded, document.extracted, document.review.required.
  • State: document_id, document_type, confidence, workflow_ref.
  • Output: normalized fields, extracted summary, evidence packet, and source links.

FW-05

Provider dashboard and operational intelligence

Covers coding yield, clean claim rate, denials, MLR, MIPS, CMS stars, predictive UM episodes, spend reduction, and next-best actions.

unfold_more

Backend steps

  1. Ingest claims, auth, quality, utilization, payment, pharmacy, and member engagement facts into analytics models.
  2. Calculate KPI tiles by provider, region, payer, plan, episode, and time window.
  3. Run predictive UM scoring to estimate episode volume, spend reduction, and MLR impact.

Services and metrics

Covers claim intake, CPT/ICD selection, CAC, QA review, missing attachments, clean-claim readiness, and revenue leakage checks.

unfold_more

Backend steps

Suggested Service Boundaries

IAM Service: auth, role policy, session context, tenant policy
Workflow Service: orchestration, retries, task state, correlation IDs
AI Action Service: prompt interpretation, extraction, summaries, workflow launch
Claims / Auth / Billing / Rx / Appointment / Enrollment services: domain ownership and business rules
Integration Service: payer APIs, pharmacy APIs, calendars, payment gateway, notifications
Audit and Observability Service: logs, traces, events, support replay, compliance evidence

State and Persistence Checklist

  • Draft state: enrollment, prior auth, claims, payments, follow-up configuration.
  • Operational state: claim queue, denial queue, auth queue, payment exceptions, Rx exceptions, appointment queue.
  • AI state: prompt history, file references, transcript summary, handoff packet, confidence score.
  • User-facing state: portal notifications, success messages, calendar projection, statement availability, receipts.
  • Audit state: actor, timestamp, before/after status, external request ID, workflow correlation ID.

Implementation Notes For Dev Teams

  • Use workflow IDs and correlation IDs across UI, APIs, workers, queues, and notifications.
  • Keep AI execution separate from core domain mutation so human override stays possible.
  • Treat voice and file-based actions as alternate input channels for the same backend commands.
  • Persist normalized view models for dashboards so member and provider screens load quickly without replaying every integration call.
  • Design every workflow with status + retry + escalation + audit as first-class concerns.
